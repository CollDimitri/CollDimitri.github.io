Projet professionnel · Alternance SNCF Voyageurs

Déploiement de postes kiosques
pour les contrôleurs SNCF.

Conception et déploiement de PC génériques sécurisés permettant aux contrôleurs de consulter leurs horaires de service. Autologon, mode kiosque, script de heartbeat, verrouillage complet du poste — tout automatisé, sans intervention humaine au démarrage.

ContexteAlternance SNCF Voyageurs
Durée2 semaines
EnvironnementWindows 10 LTSC · AD
Postes déployés8 kiosques

1. Contexte et besoin

Dans les locaux techniques de SNCF Voyageurs, les contrôleurs de train ont besoin de consulter leurs horaires et missions de service avant chaque prise de poste. Ces informations sont disponibles sur un portail web interne. Le problème historique : les postes partagés n'étaient pas configurés de façon homogène, les sessions n'étaient pas fermées correctement, et des utilisateurs non autorisés pouvaient modifier les réglages.

On m'a confié la mission de déployer 8 postes "kiosques" identiques avec les contraintes suivantes :

  • Le PC démarre seul et ouvre automatiquement le portail horaires, sans que l'agent touche au clavier.
  • L'interface est verrouillée : pas d'accès au bureau, à l'explorateur ou aux paramètres.
  • En cas de coupure de courant, le poste doit redémarrer et reprendre automatiquement.
  • Un log de démarrage est envoyé au serveur de supervision pour savoir quel poste est opérationnel.
  • Tous les postes doivent être strictement identiques (même image disque, même configuration).

2. Architecture de la solution

3. Étape 1 — Création du compte kiosque local

Chaque poste dispose d'un compte local dédié kiosque, créé via PowerShell pour garantir l'homogénéité :

# Création du compte local kiosque avec mot de passe non-expirant $pwd = ConvertTo-SecureString "K1osque!SNCF#2025" -AsPlainText -Force New-LocalUser -Name "kiosque" ` -Password $pwd ` -FullName "Poste Kiosque Contrôleurs" ` -Description "Compte auto-login - NE PAS SUPPRIMER" ` -PasswordNeverExpires ` -UserMayNotChangePassword # On NE l'ajoute PAS aux administrateurs locaux # Groupe Users uniquement = droits minimaux

4. Étape 2 — Autologon via le registre Windows

L'autologon permet au PC de se connecter automatiquement avec le compte kiosque au démarrage, sans qu'aucun agent tape un mot de passe. La configuration passe par la clé de registre Winlogon :

# Chemin de registre concerné : # HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Winlogon $regPath = "HKLM:\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Winlogon" Set-ItemProperty -Path $regPath -Name "AutoAdminLogon" -Value "1" Set-ItemProperty -Path $regPath -Name "DefaultUserName" -Value "kiosque" Set-ItemProperty -Path $regPath -Name "DefaultPassword" -Value "K1osque!SNCF#2025" Set-ItemProperty -Path $regPath -Name "DefaultDomainName" -Value "." # "." = machine locale (pas de domaine pour ce compte) # Désactiver l'écran de verrouillage Set-ItemProperty -Path $regPath -Name "DisableLockWorkstation" -Value "1"
Sécurité — Le mot de passe stocké en clair dans le registre est lisible par un admin local. Dans un environnement critique, on utilise plutôt l'outil Autologon.exe de Sysinternals (Microsoft), qui chiffre le mot de passe dans LSA Secrets : les données sont illisibles sans être admin de domaine. J'ai utilisé cette version sur les postes finaux.

5. Étape 3 — Shell replacement : Chrome comme bureau

Plutôt qu'un bureau Windows classique, le shell du compte kiosque est remplacé directement par Chrome. Le résultat : à la connexion, aucun bureau ne s'affiche, Chrome s'ouvre directement en plein écran sur le portail horaires.

# Dans Winlogon, on remplace "explorer.exe" par Chrome en mode kiosque $chromePath = "C:\Program Files\Google\Chrome\Application\chrome.exe" $url = "https://horaires.intranet.sncf.fr" $shellCmd = "$chromePath --kiosk $url `` --disable-infobars `` --disable-session-crashed-bubble `` --disable-restore-session-state `` --no-first-run `` --disable-features=TranslateUI `` --disable-pinch `` --overscroll-history-navigation=0" Set-ItemProperty -Path $regPath -Name "Shell" -Value $shellCmd

Avec ce réglage, même si l'agent appuie sur Échap ou F11, Chrome reste en plein écran et il n'y a aucun bureau accessible derrière. Le seul moyen de quitter : fermer Chrome via le Gestionnaire des tâches… qui est désactivé (voir étape suivante).

6. Étape 4 — Verrouillage via stratégies locales

J'ai configuré les stratégies de groupe locales (gpedit.msc) pour verrouiller complètement le poste :

Pour renforcer encore, j'ai aussi configuré ces valeurs de registre :

$polPath = "HKCU:\Software\Microsoft\Windows\CurrentVersion\Policies" # Désactiver le Gestionnaire des tâches New-Item -Path "$polPath\System" -Force | Out-Null Set-ItemProperty -Path "$polPath\System" -Name "DisableTaskMgr" -Value 1 # Désactiver le bouton Windows (clé Windows du clavier) Set-ItemProperty -Path "HKCU:\Software\Microsoft\Windows\CurrentVersion\Policies\Explorer" ` -Name "NoWinKeys" -Value 1 # Empêcher tout changement de fond d'écran Set-ItemProperty -Path "$polPath\ActiveDesktop" -Name "NoChangingWallPaper" -Value 1

7. Étape 5 — Script de heartbeat (log de démarrage)

À chaque démarrage du poste, un script PowerShell envoie une requête HTTP au serveur de monitoring interne. Cela permet de savoir en temps réel quels kiosques sont allumés et opérationnels, sans avoir à aller vérifier physiquement.

# kiosk-heartbeat.ps1 — lancé au démarrage de session via le Planificateur de tâches $hostname = $env:COMPUTERNAME $timestamp = Get-Date -Format "yyyy-MM-dd HH:mm:ss" $ip = (Get-NetIPAddress -AddressFamily IPv4 -InterfaceAlias "Ethernet").IPAddress $monUrl = "https://monitoring.intranet.sncf.fr/api/kiosk-log" $body = @{ host = $hostname ip = $ip timestamp = $timestamp status = "online" } | ConvertTo-Json try { Invoke-RestMethod -Uri $monUrl -Method POST -Body $body -ContentType "application/json" Write-EventLog -LogName Application -Source "KioskHeartbeat" ` -EntryType Information -EventId 1000 ` -Message "Heartbeat envoyé : $hostname ($ip)" } catch { # En cas d'échec réseau, on log localement et on réessaie dans 5 min Write-EventLog -LogName Application -Source "KioskHeartbeat" ` -EntryType Warning -EventId 1001 ` -Message "Échec heartbeat : $_" Start-Sleep -Seconds 300 # Re-tentative silencieuse Invoke-RestMethod -Uri $monUrl -Method POST -Body $body -ContentType "application/json" ` -ErrorAction SilentlyContinue }

Le script est enregistré dans le Planificateur de tâches (taskschd.msc) avec les paramètres suivants : Déclencheur : À l'ouverture de session de l'utilisateur "kiosque" · Exécuter avec les autorisations les plus élevées : NON (le compte kiosque n'est pas admin) · Cacher la fenêtre : OUI.

8. Étape 6 — Watchdog Chrome (redémarrage automatique)

Si Chrome plante ou se ferme pour une raison quelconque, un second script (watchdog) tourne en arrière-plan et le relance automatiquement :

# kiosk-watchdog.ps1 — tourne en boucle en arrière-plan $chromePath = "C:\Program Files\Google\Chrome\Application\chrome.exe" $url = "https://horaires.intranet.sncf.fr" $args = "--kiosk $url --disable-infobars --no-first-run" while ($true) { $proc = Get-Process -Name "chrome" -ErrorAction SilentlyContinue if (-not $proc) { Write-EventLog -LogName Application -Source "KioskWatchdog" ` -EntryType Warning -EventId 2000 ` -Message "Chrome non détecté. Relancement..." Start-Process -FilePath $chromePath -ArgumentList $args Start-Sleep -Seconds 5 } Start-Sleep -Seconds 10 # vérification toutes les 10 secondes }

9. Étape 7 — Redémarrage automatique après coupure

Configuration BIOS/UEFI sur chaque machine : dans les options Power Management, réglage de "Restore on AC Power Loss" sur Power On. Ainsi, si une coupure électrique éteint le poste, il se rallume seul au retour du courant — sans intervention.

Côté Windows, la connexion automatique se fait via l'autologon configuré précédemment, et les scripts reprennent automatiquement via le Planificateur de tâches.

10. Déploiement sur les 8 postes

Une fois le premier poste configuré et validé, j'ai créé un script de déploiement maître qui applique toute la configuration en une seule exécution. Pour les 7 postes restants, j'ai utilisé deux approches combinées :

  • Clonage disque : image Clonezilla du premier poste déployée sur les suivants via une clé USB bootable (les postes étant identiques matériellement).
  • Post-configuration : un script PowerShell s'exécute au premier démarrage pour personnaliser le nom du poste (KIOSK-01 à KIOSK-08) et mettre à jour le hostname dans le script de heartbeat.

Durée totale pour les 8 postes : environ 4 heures.

11. Tests & validation

  • Redémarrage à froid → Chrome s'ouvre seul sur le portail ✔
  • Coupure secteur simulée → reprise automatique ✔
  • Tentative d'accès au bureau (Ctrl+Alt+Suppr) → bloqué ✔
  • Plantage Chrome simulé (kill process) → watchdog relance en 10 secondes ✔
  • Dashboard monitoring → tous les kiosques remontent bien au démarrage ✔
  • Tentative de changement de page via la barre d'adresse → Chrome kiosque ne l'affiche pas ✔
Résultat final — Les 8 kiosques sont opérationnels en production. Le responsable informatique peut suivre leur état depuis le dashboard de monitoring. Aucune intervention manuelle n'est nécessaire après une coupure. Les agents contrôleurs n'ont pas besoin de toucher à la configuration.
